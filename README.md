# CyberSec Drive Hierarchy Standard (CDHS)

## 1. Вступ (Introduction)
Цей документ описує **CyberSec Drive Hierarchy Standard (CDHS)** — уніфікований стандарт організації файлового простору для фахівців з кібербезпеки.

### 1.1. Призначення стандарту

- **Уникнення хаосу та стандартизація.**
- **Швидка орієнтація.**
- **Безпека даних та контроль доступу.**
    

### 1.2. Область застосування
Стандарт CDHS можна використовувати незалежно від платформи зберігання. Його принципи та структура застосовуються до будь-яких спільних файлових систем:

- Хмарні сховища (Google Workspace Shared Drives, Microsoft SharePoint, OneDrive).
- Локальні мережеві сховища (NAS, файлові сервери SMB/NFS).
- Внутрішні системи синхронізації (Nextcloud, ownCloud).
    

### 1.3. Філософія CDHS
Архітектура CDHS надихається ідеологією **Linux Filesystem Hierarchy Standard (FHS)**.

Принципи стандарту:
- **Наслідування ідей Linux FHS.**
- **Числова префіксація:** Усі кореневі та підпорядковані теки використовують числовий префікс (наприклад, `00_`, `01_`). Це гарантує жорстко закріплений порядок сортування незалежно від налаштувань операційної системи чи хмарного клієнта користувача.
- **Ізоляція даних:** Дані групуються за їхнім типом, життєвим циклом та вимогами до конфіденційності.

## 2. Структура верхнього рівня (Top-Level Hierarchy)
Щоб уникнути змішування статичної нормативної бази, динамічних операційних даних та ізольованих проєктів, стандарт визначає 10 базових директорій.

Числова префіксація (від `00` до `99`) гарантує незмінний порядок відображення тек незалежно від платформи чи сортування за алфавітом.

### 2.1. Візуальна структура (Directory Tree)

```Plaintext
/ (Кореневий каталог підрозділу КБ)
├── 00_GOVERNANCE/          # Нормативна база та політики
├── 01_PROJECTS/            # Ізольовані робочі проєкти та аудити
├── 02_OPERATIONS/          # Операційна діяльність та інциденти
├── 03_SECURITY_MONITORING/ # Логи, SIEM-вивантаження, IOC
├── 04_DOCUMENTATION/       # Архітектура, схеми мереж, інвентаризація
├── 05_TRAINING/            # Навчальні матеріали та сертифікації
├── 06_TEMPLATES/           # Шаблони документів та звітів
├── 07_RESEARCH/            # Дослідження, R&D, тестування інструментів
├── 08_ARCHIVE/             # Сховище закритих та неактуальних даних
└── 99_PERSONAL/            # Особистий робочий простір
```

### 2.2. Опис каталогів
Кожна директорія верхнього рівня має своє суворе призначення.

| **Директорія**               | **Опис та призначення**                                                                                                                                                | **Доступ (Базовий)**   |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------- |
| **`00_GOVERNANCE`**          | **Статичні та затверджені документи.** Містить загальносистемні конфігурації: політики безпеки, стандарти, процедури (SOP), оцінки ризиків.  | Read-only              |
| **`01_PROJECTS`**            | **Ізольовані сутності.**Кожен пентест, аудит чи велике впровадження живе тут у власній ізольованій теці зі своїм циклом.   | Обмежений (по проєкту) |
| **`02_OPERATIONS`**          | **Динамічні (змінні) дані.** Щоденна рутина: управління вразливістями, реагування на інциденти (IR), процеси управління змінами (Change Management).                   | Write (Тільки КБ)      |
| **`03_SECURITY_MONITORING`** | **Журналювання та моніторинг.** Сирі дані, вивантаження з SIEM, аналіз логів, дашборди, Threat Intelligence та індикатори компрометації (IOC).                         | Обмежений              |
| **`04_DOCUMENTATION`**       | **Технічна документація систем.** Топології мереж, описи архітектури, інвентаризація активів (Asset Inventory). Інформація про те, _як_ побудована інфраструктура.     | Read/Write             |
| **`05_TRAINING`**            | **Розвиток компетенцій.** Внутрішні курси, матеріали для сертифікацій, записи воркшопів та тренінгів для команди КБ.                                                   | Read/Write             |
| **`06_TEMPLATES`**           | **Спільні ресурси (Shared Data).** Архітектурно-незалежні дані: бланки, шаблони звітів, стандартизовані чек-лісти, базові форми.                                       | Read-only              |
| **`07_RESEARCH`**            | **Пісочниця (Sandbox).** R&D простір для аналізу нових інструментів, тестування концептів (PoC) або дослідження нових векторів атак.                                   | Write                  |
| **`08_ARCHIVE`**             | **Холодне зберігання.** Усе, що втратило операційну актуальність, але має зберігатися для історії чи комплаєнсу (завершені проєкти, старі політики).                   | Read-only              |
| **`99_PERSONAL`**            | **Домашні директорії.** Ізольований простір (наприклад, `99_PERSONAL/d.holub`) для чорнових нотаток та робочих файлів, які ще не готові до переміщення у спільні теки. | Повний (для власника)  |
