## 3. Специфікація каталогів (Directory Specifications)

Цей розділ деталізує внутрішню структуру кореневих директорій, їх призначення та правила роботи з файлами всередині них.

### 3.1. `00_GOVERNANCE` (Нормативна база)

Каталог призначений **виключно для затверджених документів**. Це єдине джерело правди (Single Source of Truth) для політик та регламентів.

```lua
00_GOVERNANCE/
 ├── 01_Policies/           # Політики (напр., Загальна політика ІБ)
 ├── 02_Standards/          # Стандарти, регламенти чи додатки до політик
 ├── 03_Procedures/         # Покрокові процедури (SOPs)
 ├── 04_Guidelines/         # Рекомендації та найкращі практики
 ├── 05_Risk_Assessments/   # Реєстри ризиків та звіти з оцінки
 └── 06_Compliance/         # Звіти для аудиторів, сертифікати, тощо
```

**Правила затвердження документів:**

1. Розробка нового документа ведеться в особистій теці автора або у відповідному проєкті (`01_PROJECTS`).
2. Документ переміщується до `00_GOVERNANCE` **тільки після його офіційного затвердження** керівництвом.
3. Усі файли тут мають статус **Read-only** для всієї команди, окрім відповідального за ведення документації (або адміністраторів файлового простору).

### 3.2. `01_PROJECTS` (Проєкти та Аудити)

Інформація щодо об'ємних у часі завдань: внутрішніх аудитів, зовнішніх пентестів, впровадження нових систем захисту.

```lua
01_PROJECTS/
 ├── Active/
 │    ├── 2026_ClientA_Pentest/
 │    │     ├── 01_Scope/           # Договори, межі тестування, правила взаємодії (RoE)
 │    │     ├── 02_Recon/           # OSINT, результати розвідки
 │    │     ├── 03_Scans/           # Сирі результати Nmap, Nessus, Burp Suite
 │    │     ├── 04_Exploitation/    # Скрипти, докази експлуатації (PoC), скріншоти
 │    │     └── 05_Report/          # Чернетки та фінальні версії звітів
 │    └── 2026_SIEM_Deployment/
 └── Completed/
```

**Стандарт ведення проєктів:**

- Назва теки проєкту має містити рік, місяць та суть: `YYYY-MM_Project_Name`.
- Після підписання актів або фінальної презентації проєкту, уся тека з `Active/` цілком переміщується до `Completed/`, а права змінюються на Read-only.

### 3.3. `02_OPERATIONS` & `03_SECURITY_MONITORING`

Ці дві директорії покривають щоденну операційно-аналітичну рутину.

**`02_OPERATIONS` (Операційна діяльність):**

```lua
02_OPERATIONS/
 ├── Incident_Response/       # Теки під кожен інцидент (напр., 2026-02-28_Malware_Host_X)
 ├── Change_Management/       # Оцінка ризиків змін, робочі чернетки нових конфігів
 ├── Access_Management/       # Матриці доступу, реєстри привілейованих користувачів
 └── Vulnerability_Management/# Періодичні сирі звіти сканування інфраструктури
```

**`03_SECURITY_MONITORING` (Моніторинг):**

```lua
03_SECURITY_MONITORING/
 ├── SIEM_Exports/            # Вивантаження логів для ретроспективного аналізу
 ├── Log_Analysis/            # Робочі файли аналізу логів
 ├── Threat_Intelligence/     Аналітика загроз (звіти про хакерські атаки, нові вразливості, тренди)
 ├── Alerts/                  # Статистика спрацювань систем (IDS/IPS/EDR)
 └── IOC/                     # Індикатори компрометації (списки IP, хешів, доменів)
```

### 3.4. `04_DOCUMENTATION` & `05_TRAINING`

Знання про інфраструктуру та навчальні матеріали.

**`04_DOCUMENTATION` (Документація інфраструктури):**

```lua
04_DOCUMENTATION/
 ├── Network_Diagrams/        # Схеми L2/L3, архітектура зон безпеки (DMZ)
 ├── Asset_Inventory/         # Реєстри активів (сервери, сервіси, ПЗ)
 ├── System_Descriptions/     # Робоча документація по обладнанню чи системах захисту
 └── Architecture/            # High-Level та Low-Level Design (HLD/LLD)
```

**`05_TRAINING` (Навчання):**

```lua
05_TRAINING/
 ├── Internal/                # Воркшопи, записи мітів
 ├── Certifications/          # Матеріали для підготовки (CEH, CISSP, OSCP тощо)
 ├── Courses/                 # Придбані курси або загальнодоступні лекції
 └── Materials/               # Книги, cheat sheets, статті
```

### 3.5. `06_TEMPLATES`, `07_TOOLS_AND_SCRIPTS` & `08_ARCHIVE`

**`06_TEMPLATES` (Шаблони - Read-only):** Еталонні шаблони файлів/документів (використовуються для копіювання).

```lua
06_TEMPLATES/
 ├── Report_Templates/        # Шаблони пентест-звітів, аудитів, звітів про інциденти
 ├── Policy_Templates/        # Бланки для написання нових політик
 ├── Checklists/              # Чек-лісти (напр., Hardening_Linux_Checklist.xlsx)
 └── Forms/                   # Форми запитів (доступи, фаєрволи)
```

**`07_TOOLS_AND_SCRIPTS` (Інструментарій):** Локальний репозиторій утиліт та робочих файлів для автоматизації.

```lua
07_TOOLS_AND_SCRIPTS/
 ├── Automation_Scripts/      # Скрипти (bash, python, powershell) для бекапів, парсингу
 ├── Pentest_Payloads/        # Кастомні пейлоади для тестувань
 ├── Dictionaries/            # Словники (wordlists)
 ├── Config_Baselines/        # Еталонні конфіг-файли (hardening configs для Linux/Windows)
 └── Software_Distributions/  # Специфічне ПЗ або портативні версії утиліт
```

**`08_ARCHIVE` (Архів - Read-only):** Холодне сховище. Сюди інформація потрапляє згідно з політикою утримання даних.

```lua
08_ARCHIVE/
 ├── Projects/                # Проєкти з 01_PROJECTS/Completed, яким більше 1-2 років
 ├── Policies_Old/            # Попередні версії політик (для аудиту історії)
 └── Logs_Old/                # Старі вивантаження логів
```

### 3.6. `99_PERSONAL` (Особистий простір)

Директорія для індивідуальної роботи. Як простір може існувати окремо від загальної робочої зони, тому її створення опціональне.

```lua
99_PERSONAL/
 ├── user.name2/              # Робочий простір спеціаліста
 └── user.name3/
```

**Правила використання `99_PERSONAL`:**

1. **Повний контроль:** Власник теки має повні права на створення, зміну та видалення файлів всередині своєї директорії.  
2. **Чернетки:** Це місце для "брудних" розрахунків, чорнових звітів, локальних скриптів.
3. **Заборона шарингу:** Забороняється надавати посилання іншим на файли, що лежать у `99_PERSONAL`. Якщо документ готовий для спільної роботи чи передачі — він має бути переміщений у відповідну цільову теку (наприклад, у `01_PROJECTS` або `00_GOVERNANCE`).
