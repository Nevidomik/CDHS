## 4. Модель управління доступом (Access Control Model)
Оскільки базовою інфраструктурою для CDHS є спільні диски (Shared Drives) Google Workspace, модель управління доступом (RBAC) тісно інтегрована з нативними ролями платформи.

### 4.1. Базові рівні доступу (Google Drive Roles)
У стандарті використовуються такі рівні прав:

- **[M] Manage (Керуючий / Власник контенту):** Повний контроль. Створення, видалення, переміщення файлів та тек, налаштування доступів.   
- **[W] Write (Учасник / Редактор):** Робочий рівень. Додавання нових файлів та редагування існуючих, налаштування доступів.
- **[C] Comment (Коментатор):** Рівень рев'ювера. Читання файлів та залишення коментарів/пропозицій без можливості прямого редагування чи видалення.
- **[R] Read (Читач):** Тільки перегляд та завантаження.
- **[-] None:** Доступ відсутній (папка не відображається або заблокована).

### 4.2. Ролі користувачів та Матриця доступу
У таблиці нижче наведено базові права для кожної ролі на рівні кореневих директорій.

| **Директорія**               | **Керівник (CISO)** | **Спеціаліст** | **Аудитор** | **Суміжні IT відділи** |
| ---------------------------- | ------------------- | -------------- | ----------- | ---------------------- |
| **`00_GOVERNANCE`**          | M                   | M              | C           | R                      |
| **`01_PROJECTS`**            | M                   | M              | C**         | -                      |
| **`02_OPERATIONS`**          | M                   | W              | -           | -                      |
| **`03_SECURITY_MONITORING`** | M                   | W              | R           | -                      |
| **`04_DOCUMENTATION`**       | M                   | M              | R           | C**                    |
| **`05_TRAINING`**            | M                   | W              | -           | -                      |
| **`06_TEMPLATES`**           | M                   | M              | -           | R                      |
| **`07_RESEARCH`**            | M                   | W              | -           | -                      |
| **`08_ARCHIVE`**             | M                   | M              | R           | -                      |
| **`99_PERSONAL`**            | M                   | M (Своя)       | -           | -                      |

_* Спеціалісти мають право коментувати (C) затверджені політики для пропозицій щодо покращення. Редагування (W) можливе лише в робочих теках чорновиків, перед перенесенням у корінь._
_** Доступ надається точково. Наприклад, системні адміністратори можуть коментувати (C) архітектурні схеми в `04_DOCUMENTATION`, а аудитори — залишати коментарі до фінальних звітів у `01_PROJECTS`._

### 4.3. Принцип "Need-to-know" (Мінімальні привілеї)
Принцип "Need-to-know" означає, що доступ до чутливих даних надається виключно тим співробітникам, яким це необхідно для виконання поточного робочого завдання.

**Практичне застосування в CDHS:**
1. **Ізоляція інцидентів (`02_OPERATIONS/Incident_Response`):** При розслідуванні критичного інциденту  тека з матеріалами розслідування розшарюється **виключно** для фахівців, які безпосередньо залучені до Incident Response Team.
2. **Чутливі проєкти (`01_PROJECTS/Active`):** Якщо проводиться Red Teaming або тестування на проникнення систем з найвищим рівнем критичності, доступ до теки проєкту `YYYY_ProjectName` обмежується складом конкретної робочої групи.
3. **Реалізація на Google Drive:**
  - **Базове обмеження на кореневому рівні:** На рівні кореневих директорій встановлюється мінімально необхідний доступ — **Читач (R)** або **Коментатор (C)** для всієї команди.
  - **Точкове підвищення привілеїв:** Права на редагування (**W**) або управління (**M**) призначаються безпосередньо на цільові вкладені теки (наприклад, `01_PROJECTS/Active/2026_ObjectX`) або навіть на окремі файли всередині них (наприклад, робочий драфт звіту).
