# CyberSec Drive Hierarchy Standard (CDHS)

## 1. Вступ (Introduction)
Цей документ описує **CyberSec Drive Hierarchy Standard (CDHS)** — уніфікований стандарт організації файлового простору для фахівців з кібербезпеки.

### 1.1. Призначення стандарту

- **Уникнення хаосу та стандартизація.**
- **Швидка орієнтація.**
- **Безпека даних та контроль доступу.**
    

### 1.2. Область застосування
Стандарт CDHS можна використовувати незалежно від платформи зберігання. Його принципи та структура застосовуються до будь-яких спільних файлових систем:

- Хмарні сховища (Google Workspace Shared Drives, Microsoft SharePoint, OneDrive).
- Локальні мережеві сховища (NAS, файлові сервери SMB/NFS).
- Внутрішні системи синхронізації (Nextcloud, ownCloud).
    

### 1.3. Філософія CDHS
Архітектура CDHS надихається ідеологією **Linux Filesystem Hierarchy Standard (FHS)**.

Принципи стандарту:
- **Наслідування ідей Linux FHS.**
- **Числова префіксація:** Усі кореневі та підпорядковані теки використовують числовий префікс (наприклад, `00_`, `01_`). Це гарантує жорстко закріплений порядок сортування незалежно від налаштувань операційної системи чи хмарного клієнта користувача.
- **Ізоляція даних:** Дані групуються за їхнім типом, життєвим циклом та вимогами до конфіденційності.

## 2. Структура верхнього рівня (Top-Level Hierarchy)
Щоб уникнути змішування статичної нормативної бази, динамічних операційних даних та ізольованих проєктів, стандарт визначає 10 базових директорій.

Числова префіксація (від `00` до `99`) гарантує незмінний порядок відображення тек незалежно від платформи чи сортування за алфавітом.

### 2.1. Візуальна структура (Directory Tree)

```lua
/ (Кореневий каталог підрозділу КБ)
├── 00_GOVERNANCE/          # Нормативна база та політики
├── 01_PROJECTS/            # Ізольовані робочі проєкти та аудити
├── 02_OPERATIONS/          # Операційна діяльність та інциденти
├── 03_SECURITY_MONITORING/ # Логи, SIEM-вивантаження, IOC
├── 04_DOCUMENTATION/       # Архітектура, схеми мереж, інвентаризація
├── 05_TRAINING/            # Навчальні матеріали та сертифікації
├── 06_TEMPLATES/           # Шаблони документів та звітів
├── 07_RESEARCH/            # Дослідження, R&D, тестування інструментів
├── 08_ARCHIVE/             # Сховище закритих та неактуальних даних
└── 99_PERSONAL/            # Особистий робочий простір
```

### 2.2. Опис каталогів
Кожна директорія верхнього рівня має своє суворе призначення.

| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------- |

| **`01_PROJECTS`**            | **Ізольовані сутності.**Кожен пентест, аудит чи велике впровадження живе тут у власній ізольованій теці зі своїм циклом.   | Обмежений (по проєкту) |

## 3. Специфікація каталогів (Directory Specifications)
Цей розділ деталізує внутрішню структуру кожної кореневої теки, її призначення та правила роботи з файлами всередині неї.

### 3.1. `00_GOVERNANCE` (Нормативна база)
Каталог призначений **виключно для затверджених документів**, які регламентують роботу відділу та компанії в розрізі ІБ. Це єдине джерело правди (Single Source of Truth) для політик та стандартів.

```lua
00_GOVERNANCE/
 ├── 01_Policies/           # Політики (напр., Загальна політика ІБ)
 ├── 02_Standards/          # Стандарти, регламенти чи додатки до політик
 ├── 03_Procedures/         # Покрокові процедури (SOPs)
 ├── 04_Guidelines/         # Рекомендації та найкращі практики
 ├── 05_Risk_Assessments/   # Реєстри ризиків та звіти з оцінки
 └── 06_Compliance/         # Документація для відповідності (ISO 27001, GDPR тощо)
```

**Правила затвердження документів:**
1. Розробка (Drafting) нового документа ведеться в особистій теці автора (`99_PERSONAL`) або у відповідному проєкті (`01_PROJECTS`).
2. Документ переміщується до `00_GOVERNANCE` **тільки після його офіційного затвердження** керівництвом.
3. Усі файли тут мають статус **Read-only** для всієї команди, окрім відповідального за ведення документації (або керівника).    

### 3.2. `01_PROJECTS` (Проєкти та Аудити)
Тут зберігається інформація щодо обмежених у часі завдань: внутрішніх аудитів, зовнішніх пентестів, впровадження нових систем захисту.

```lua
01_PROJECTS/
 ├── Active/
 │    ├── 2026_ClientA_Pentest/
 │    │     ├── 01_Scope/           # Договори, межі тестування, правила взаємодії (RoE)
 │    │     ├── 02_Recon/           # OSINT, результати розвідки
 │    │     ├── 03_Scans/           # Сирі результати Nmap, Nessus, Burp Suite
 │    │     ├── 04_Exploitation/    # Скрипти, докази експлуатації (PoC), скріншоти
 │    │     ├── 05_Report/          # Чернетки та фінальні версії звітів
 │    │     └── 99_Deliverables/    # Фінальні артефакти, передані замовнику/керівництву
 │    └── 2026_SIEM_Deployment/
 └── Completed/
```

**Стандарт ведення проєктів:**
- Кожен проєкт створюється за єдиним шаблоном (див. `06_TEMPLATES`).
- Назва теки проєкту має містити рік та суть: `YYYY_Project_Name`.
    
- **Правило перенесення:** Після підписання актів або фінальної презентації проєкту, уся тека з `Active/` цілком переміщується до `Completed/`, а права змінюються на Read-only.

### 3.3. `02_OPERATIONS` & `03_SECURITY_MONITORING`
Ці дві директорії покривають щоденну інженерну рутину.

**`02_OPERATIONS` (Операційна діяльність):**

```lua
02_OPERATIONS/
 ├── Incident_Response/       # Теки під кожен інцидент (напр., 2026-02-28_Malware_Host_X)
 ├── Change_Management/       # Запити на зміни конфігурацій
 ├── Access_Management/       # Матриці доступу, заявки на надання прав
 ├── Vulnerability_Management/# Періодичні звіти сканування інфраструктури
 └── Backups/                 # Конфіги безпекових засобів (Firewalls, WAF)
```

**`03_SECURITY_MONITORING` (Моніторинг):**

```lua
03_SECURITY_MONITORING/
 ├── SIEM_Exports/            # Вивантаження логів для ретроспективного аналізу
 ├── Log_Analysis/            # Робочі файли аналізу логів
 ├── Threat_Intelligence/     # Файди, звіти APT-угруповань
 ├── Alerts/                  # Статистика спрацювань систем (IDS/IPS/EDR)
 └── IOC/                     # Індикатори компрометації (списки IP, хешів, доменів)
```

### 3.4. `04_DOCUMENTATION` & `05_TRAINING`
Знання про інфраструктуру та знання команди.

**`04_DOCUMENTATION` (Документація інфраструктури):**

```lua
04_DOCUMENTATION/
 ├── Network_Diagrams/        # Схеми L2/L3, архітектура зон безпеки (DMZ)
 ├── Asset_Inventory/         # Реєстри активів (сервери, сервіси, ПЗ)
 ├── System_Descriptions/     # Описи налаштувань критичних систем
 └── Architecture/            # High-Level та Low-Level Design (HLD/LLD)
```

**`05_TRAINING` (Навчання):**

```lua
05_TRAINING/
 ├── Internal/                # Воркшопи, записи мітів з обміном досвідом
 ├── Certifications/          # Матеріали для підготовки (CEH, CISSP, OSCP тощо)
 ├── Courses/                 # Придбані курси або загальнодоступні лекції
 └── Materials/               # Книги, cheat sheets, статті
```

### 3.5. `06_TEMPLATES` & `08_ARCHIVE`

**`06_TEMPLATES` (Шаблони - Read-only):** Слугує еталоном для створення нових сутностей. Заборонено редагувати файли безпосередньо тут (їх потрібно копіювати).

```lua
06_TEMPLATES/
 ├── Report_Templates/        # Шаблони пентест-звітів, аудитів, звітів про інциденти
 ├── Policy_Templates/        # Бланки для написання нових політик
 ├── Checklists/              # Чек-лісти (напр., Hardening_Linux_Checklist.xlsx)
 ├── Scripts/                 # Стандартизовані скрипти для автоматизації (bash/python/powershell)
 └── Forms/                   # Форми запитів (доступи, фаєрволи)
```

**`08_ARCHIVE` (Архів - Read-only):** Холодне сховище. Сюди інформація потрапляє згідно з політикою утримання даних (Data Retention).

```lua
08_ARCHIVE/
 ├── Projects/                # Проєкти з 01_PROJECTS/Completed, яким більше 1-2 років
 ├── Policies_Old/            # Попередні версії політик (для аудиту історії)
 └── Logs_Old/                # Старі вивантаження логів
```

### 3.6. `99_PERSONAL` (Особистий простір)
Директорія для індивідуальної роботи.

```lua
99_PERSONAL/
 ├── user.name2/              # Робочий простір спеціаліста
 └── user.name3/
```

**Правила використання `99_PERSONAL`:**
1. **Повний контроль:** Власник теки має повні права на створення, зміну та видалення файлів всередині своєї директорії.  
2. **Чернетки:** Це місце для "брудних" розрахунків, чорнових звітів, локальних скриптів.
3. **Заборона шарингу:** Забороняється надавати посилання іншим на файли, що лежать у `99_PERSONAL`. Якщо документ готовий для спільної роботи чи передачі — він має бути переміщений у відповідну цільову теку (наприклад, у `01_PROJECTS` або `00_GOVERNANCE`).

## 4. Модель управління доступом (Access Control Model)
Оскільки базовою інфраструктурою для CDHS є спільні диски (Shared Drives) Google Workspace, модель управління доступом (RBAC) тісно інтегрована з нативними ролями платформи.

### 4.1. Базові рівні доступу (Google Drive Roles)
У стандарті використовуються такі рівні прав:

- **[M] Manage (Керуючий / Власник контенту):** Повний контроль. Створення, видалення, переміщення файлів та тек, налаштування доступів.   
- **[W] Write (Учасник / Редактор):** Робочий рівень. Додавання нових файлів та редагування існуючих, налаштування доступів.
- **[C] Comment (Коментатор):** Рівень рев'ювера. Читання файлів та залишення коментарів/пропозицій без можливості прямого редагування чи видалення.
- **[R] Read (Читач):** Тільки перегляд та завантаження.
- **[-] None:** Доступ відсутній (папка не відображається або заблокована).

### 4.2. Ролі користувачів та Матриця доступу
У таблиці нижче наведено базові права для кожної ролі на рівні кореневих директорій.

| **Директорія**               | **Керівник (CISO)** | **Спеціаліст** | **Аудитор** | **Суміжні IT відділи** |
| ---------------------------- | ------------------- | -------------- | ----------- | ---------------------- |
| **`00_GOVERNANCE`**          | M                   | M              | C           | R                      |
| **`01_PROJECTS`**            | M                   | M              | C**         | -                      |
| **`02_OPERATIONS`**          | M                   | W              | -           | -                      |
| **`03_SECURITY_MONITORING`** | M                   | W              | R           | -                      |
| **`04_DOCUMENTATION`**       | M                   | M              | R           | C**                    |
| **`05_TRAINING`**            | M                   | W              | -           | -                      |
| **`06_TEMPLATES`**           | M                   | M              | -           | R                      |
| **`07_RESEARCH`**            | M                   | W              | -           | -                      |
| **`08_ARCHIVE`**             | M                   | M              | R           | -                      |
| **`99_PERSONAL`**            | M                   | M (Своя)       | -           | -                      |

_* Спеціалісти мають право коментувати (C) затверджені політики для пропозицій щодо покращення. Редагування (W) можливе лише в робочих теках чорновиків, перед перенесенням у корінь._
_** Доступ надається точково. Наприклад, системні адміністратори можуть коментувати (C) архітектурні схеми в `04_DOCUMENTATION`, а аудитори — залишати коментарі до фінальних звітів у `01_PROJECTS`._

### 4.3. Принцип "Need-to-know" (Мінімальні привілеї)
Принцип "Need-to-know" означає, що доступ до чутливих даних надається виключно тим співробітникам, яким це необхідно для виконання поточного робочого завдання.

**Практичне застосування в CDHS:**
1. **Ізоляція інцидентів (`02_OPERATIONS/Incident_Response`):** При розслідуванні критичного інциденту  тека з матеріалами розслідування розшарюється **виключно** для фахівців, які безпосередньо залучені до Incident Response Team.
2. **Чутливі проєкти (`01_PROJECTS/Active`):** Якщо проводиться Red Teaming або тестування на проникнення систем з найвищим рівнем критичності, доступ до теки проєкту `YYYY_ProjectName` обмежується складом конкретної робочої групи.
3. **Реалізація на Google Drive:**
  - **Базове обмеження на кореневому рівні:** На рівні кореневих директорій встановлюється мінімально необхідний доступ — **Читач (R)** або **Коментатор (C)** для всієї команди.
  - **Точкове підвищення привілеїв:** Права на редагування (**W**) або управління (**M**) призначаються безпосередньо на цільові вкладені теки (наприклад, `01_PROJECTS/Active/2026_ObjectX`) або навіть на окремі файли всередині них (наприклад, робочий драфт звіту).

  ## 5. Конвенція іменування файлів (File Naming Conventions)
Уніфікований підхід до іменування файлів є невіддільною частиною CDHS.

### 5.1. Базові правила
1. **Формат дат (ISO 8601):** Завжди використовується формат `YYYY-MM-DD` (або `YYYY-MM` для місячних звітів) на початку назви. Це гарантує ідеальне хронологічне сортування у будь-якій файловій системі.  
2. **Заборона пробілів та спецсимволів:** Пробіли ускладнюють роботу в командному рядку та ламають URL-посилання.
    - Використовуйте нижнє підкреслення `_` для розділення логічних блоків (Дата, Проєкт, Тип).
    - Використовуйте дефіс `-` для розділення слів всередині одного блоку.
    - Заборонені символи: `! @ # $ % ^ & * ( ) + = < > ? / \ | { } [ ] ~`
        
3. **Версіонування:** Наприкінці назви (перед розширенням) вказується версія документа у форматі `vX.Y`.
    - `v0.X` — чорновик (Draft), робоча версія в процесі написання.
    - `v1.0`, `v2.0` — затверджені мажорні версії (Final).
    - `v1.1`, `v1.2` — мінорні правки до затвердженого документа (виправлення одруківок, уточнення).
        
4. **Латиниця (Рекомендація):** Для технічних артефактів, логів, скриптів та звітів, які можуть оброблятися автоматизовано, рекомендується використовувати виключно латинські літери (ASCII).

### 5.2. Формат для звітів та операційних файлів
Застосовується у директоріях `01_PROJECTS`, `02_OPERATIONS`, `03_SECURITY_MONITORING`.

## 6. Життєвий цикл даних (Data Lifecycle Management)
Управління життєвим циклом даних запобігає захаращенню спільного простору.

### 6.1. Правила створення (Creation & Drafting)
Для уникнення плутанини між "сирими" напрацюваннями та затвердженими документами діє суворе правило: жоден файл не створюється одразу в цільовій еталонній директорії (наприклад, у `00_GOVERNANCE`).

1. **Індивідуальні чернетки:** Якщо розробка документа (нової політики, інструкції, архітектурної схеми) ведеться одноосібно, файл створюється та зберігається в особистій теці автора (наприклад, `99_PERSONAL/d.holub`).
2. **Командні чернетки:** Якщо над драфтом працює група (наприклад, звіт з активного пентесту), він розміщується у відповідній підтеці проєкту `01_PROJECTS/Active/YYYY_Project_Name/05_Report/`.
3. **Публікація:** Лише після фінального рев'ю та офіційного затвердження готовий документ переміщується до відповідного розділу (наприклад, `00_GOVERNANCE/01_Policies`), де йому присвоюється фінальна версія (наприклад, `v1.0`), а права змінюються на Read-only / Коментатор.

### 6.2. Процедура архівування (Archiving Procedure)
Архівування — це регулярний процес очищення "гарячих" директорій від даних, які втратили оперативну актуальність, але необхідні для історії, розслідувань чи комплаєнсу.

1. **Архівування проєктів:** Після успішної здачі проєкту (аудиту, пентесту) він спочатку переміщується з `01_PROJECTS/Active` до `01_PROJECTS/Completed`. Раз на півроку провідний спеціаліст переносить проєкти, які перебувають у `Completed` довше 12 місяців, у холодне сховище: `08_ARCHIVE/Projects/YYYY`.
2. **Архівування нормативної бази:** При випуску нової мажорної версії політики (наприклад, `v2.0`), стара версія (`v1.0`) переміщується до `08_ARCHIVE/Policies_Old`. До назви старого файлу додається префікс `[OBSOLETE]`. Це дозволяє аудиторам відстежувати еволюцію процесів.
3. **Архівування логів та інцидентів:** Матеріали закритих інцидентів (`02_OPERATIONS/Incident_Response`) та старі вивантаження з SIEM по завершенню кварталу чи року переміщуються до `08_ARCHIVE/Logs_Old`.
4. **Заморозка прав:** Будь-яка тека чи файл, що потрапляє до `08_ARCHIVE`, автоматично переводиться у режим **Read-only** для всієї команди. Зміна історичних даних суворо заборонена.

### 6.3. Політика утримання (Retention Policy) та знищення
Дані не можуть зберігатися вічно. Залежно від нормативних вимог та внутрішніх політик ІТ-інфраструктури, визначаються терміни життя документів в архіві, після чого вони підлягають безпечному видаленню.

- **Операційні логи та сирі результати сканувань:** Зберігаються 1 рік (або згідно з вимогами законодавства/регуляторів), після чого видаляються.
- **Матеріали розслідування інцидентів (Incident Reports):** Зберігаються мінімум 3–5 років для ретроспективного аналізу (напади можуть повторюватися або бути частиною тривалих APT-кампаній).
- **Звіти з аудитів та пентестів:** Зберігаються 3 роки в архіві.
- **Життєвий цикл `99_PERSONAL`:** При звільненні спеціаліста або переході на іншу посаду, його особиста тека проходить аудит. Усі корисні скрипти чи шаблони переносяться до `05_TOOLS` або `06_TEMPLATES`, проєктні чорновики передаються наступникам, після чого сама персональна тека видаляється.
